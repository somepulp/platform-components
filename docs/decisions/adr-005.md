# ADR-005: Use HashiCorp Vault for Secrets Management

**Status:** Accepted

---

## Context

The platform requires a secrets management solution to handle sensitive configuration values — API keys, database credentials, TLS certificates, and ML model service tokens. Kubernetes provides a native `Secret` resource, but it only base64-encodes values rather than encrypting them, stores secrets in etcd without strong access control by default, and provides no audit trail of secret access. A dedicated secrets management solution is required for a production-credible platform.

---

## Decision

Deploy HashiCorp Vault as the secrets management solution for the platform, managed via `platform-components`. Vault will be the authoritative store for all sensitive values consumed by platform services and application workloads.

---

## Reasoning

HashiCorp Vault is the industry standard for secrets management in cloud-native environments. It provides a superset of what Kubernetes native Secrets offer — encrypted storage, fine-grained access control policies, dynamic secret generation, full audit logging, and multiple authentication backends including Kubernetes service account authentication.

The Kubernetes auth method is particularly relevant here: workloads (pods) authenticate to Vault using their Kubernetes service account token, and Vault returns a short-lived secret scoped to what that workload is allowed to access. This eliminates the need to pre-provision static credentials in pods — a significant security improvement over native Kubernetes Secrets.

Vault also integrates with cert-manager (chosen as the TLS certificate manager for the platform) via the Vault PKI secrets engine, enabling automated certificate issuance and rotation. This covers TLS for Harbor, Envoy Gateway, and the ML classifier service.

For the homelab context, Vault runs in development mode initially to reduce operational complexity, then transitions to a production configuration with persistent storage using the k3s local-path-provisioner.

---

## Alternatives Considered

| Option | Reason Not Chosen |
|---|---|
| Kubernetes native Secrets | Base64 encoding only, not encryption; no audit trail; no dynamic secret generation; insufficient for production-credible platform |
| Sealed Secrets (Bitnami) | Encrypts secrets at rest in Git but no dynamic secrets, no audit trail, no Kubernetes auth method — solves a narrower problem |
| External Secrets Operator + cloud provider | Requires cloud dependency (GCP Secret Manager, AWS Secrets Manager); not available for local-only setup |
| Doppler / 1Password Secrets Operator | Third-party SaaS dependency; external network requirement; not appropriate for a self-contained homelab |

---

## Consequences

**Positive:**
- Industry standard — directly relevant to production environments and ML Platform Engineer roles
- Encrypted storage with fine-grained access control policies per workload
- Kubernetes auth method eliminates static credentials in pods
- Full audit trail of all secret access — important for ML governance and compliance
- PKI secrets engine integrates with cert-manager for automated TLS certificate lifecycle
- Self-contained — no external dependencies or SaaS requirements

**Negative:**
- Operationally complex — Vault requires initialization, unsealing on restart, and careful backup of unseal keys and root token
- Single point of failure — if Vault is unavailable, workloads that depend on it for secrets cannot start
- Steeper learning curve than native Kubernetes Secrets
- Vault in development mode (initial setup) does not persist data across restarts — production configuration requires additional setup
