# ADR-007: Vault Deployment Mode and Unseal Strategy

**Status:** Accepted

---

## Context

HashiCorp Vault starts in a sealed state every time it restarts. In sealed state, Vault cannot serve secrets, authenticate clients, or perform any operations. An unseal key must be provided to decrypt Vault's storage and make it operational. This creates an operational challenge for a homelab running on a Raspberry Pi — every Pi reboot requires Vault to be manually unsealed before any dependent services (VSO, Harbor, ml-classifier) can function.

Three deployment modes were considered: manual unseal with production storage, auto-unseal with GCP KMS, and Vault dev mode.

---

## Decision

Deploy Vault in **dev mode** for the local environment. Migrate to production mode with **auto-unseal via GCP KMS** when the platform extends to GCP staging and production environments.

---

## Reasoning

### Dev mode

Vault dev mode starts pre-initialized and pre-unsealed with in-memory storage. It never requires manual unsealing — Vault is immediately operational after every restart. This eliminates the operational overhead of managing unseal keys during the learning and development phase of the homelab.

The critical limitation of dev mode is that all data is lost on restart. Every reboot of the Pi means Vault's secrets, policies, and auth configurations are gone and must be reconfigured. This is acceptable during the initial platform bootstrap phase where the primary goal is learning Vault concepts and wiring up the platform — not long-term secret persistence.

### Manual unseal with production storage

Manual unseal requires storing the unseal key securely (e.g. in a password manager) and running `vault operator unseal <key>` every time the Pi reboots. This is operationally painful for a homelab where reboots are common and the primary focus is learning rather than operations. It adds friction without meaningful security benefit at this stage.

### Auto-unseal with GCP KMS

Auto-unseal via GCP KMS is the production-grade solution. Vault retrieves the unseal key from GCP KMS on startup, decrypting itself automatically without manual intervention. This is the intended steady-state for the platform when it extends to GCP — at that point GCP infrastructure is available anyway and the cost of a KMS key is negligible.

Auto-unseal is not appropriate for the local Pi environment because it introduces a cloud dependency for a local-only cluster, defeating the purpose of a self-contained homelab.

---

## Migration Path

When the platform extends to GCP staging:

1. Provision a GCP KMS key ring and crypto key via Terraform in `platform-infra`
2. Configure Vault's `seal` stanza in `values.yaml` to use `gcpckms`
3. Initialize Vault in production mode — generates a recovery key (not an unseal key) stored securely in a password manager
4. Vault unseals automatically on every restart via GCP KMS

The migration from dev mode to production mode requires re-initializing Vault and re-configuring all secrets, policies, and auth methods — dev mode data does not migrate. This is acceptable since dev mode data is ephemeral by design.

---

## Consequences

**Positive:**
- Zero operational overhead during local development — no manual unseal steps after Pi reboots
- Vault concepts (secrets engines, auth methods, policies, VSO integration) can be learned without operational friction
- Clear migration path to production mode when GCP infrastructure is available

**Negative:**
- All Vault data lost on Pi reboot — secrets, policies, and auth configurations must be reconfigured after every restart
- Dev mode is not suitable for long-term secret persistence — any secrets stored in Vault are ephemeral
- Dev mode uses a hardcoded root token (`root`) — not suitable for production
