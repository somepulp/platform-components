# ADR-004: Use MetalLB for Bare-Metal Load Balancing

**Status:** Accepted

---

## Context

Kubernetes services of type `LoadBalancer` require an external load balancer to assign a routable IP address. In cloud environments (GKE, EKS, AKS), this is handled automatically by the cloud provider — a managed load balancer is provisioned and an external IP is assigned to the service. On bare metal (the Raspberry Pi), no such provider exists. Without a solution, `LoadBalancer` services remain in `<pending>` state indefinitely with no external IP assigned, making it impossible to expose services outside the cluster.

Envoy Gateway (chosen in [ADR-002](adr-002.md)) requires a `LoadBalancer` service to receive external traffic. Without a bare-metal load balancer, Envoy Gateway cannot function correctly.

---

## Decision

Deploy MetalLB as the bare-metal load balancer for the k3s cluster, managed via `platform-components`.

---

## Reasoning

MetalLB is the most widely adopted solution for providing `LoadBalancer` service support on bare-metal Kubernetes clusters. It watches for `LoadBalancer` services and assigns IP addresses from a configured address pool on the local network, then announces those IPs so that traffic can reach the cluster.

MetalLB supports two announcement modes:

- **Layer 2 mode** — MetalLB responds to ARP requests on the local network, making one node respond on behalf of the assigned IP. Simple to configure, no special network hardware required. Chosen for this setup.
- **BGP mode** — MetalLB peers with a BGP router to advertise routes. More scalable and production-grade, but requires a BGP-capable router. Not appropriate for a home network.

Layer 2 mode is the right choice for the Pi homelab — it requires no special network configuration and works with any standard home router.

---

## Alternatives Considered

| Option | Reason Not Chosen |
|---|---|
| NodePort services | Non-standard high ports (30000–32767); awkward for exposing HTTP/HTTPS services cleanly |
| HostNetwork | Bypasses Kubernetes networking model; security concerns; not portable |
| kube-vip | Viable alternative but primarily designed for control plane HA; MetalLB has broader service load balancing focus and larger community |
| Cloud load balancer (GCP) | Not available for local Pi environment; deferred to cloud phase |

---

## Consequences

**Positive:**
- `LoadBalancer` services work on bare metal exactly as they would in a cloud environment
- Enables Envoy Gateway to receive external traffic with a stable local IP
- Layer 2 mode requires no special network hardware or router configuration
- Widely adopted — skills transfer to any bare-metal or on-premises Kubernetes environment
- Managed via GitOps in `platform-components`

**Negative:**
- Layer 2 mode has a single-node failover limitation — only one node handles ARP for a given IP at a time. Not a concern for a single-node homelab but worth knowing for future multi-node setups
- Requires configuring an IP address pool from the local network range — needs coordination with the home router's DHCP range to avoid conflicts
- Not needed when the platform moves to GCP — cloud load balancers replace MetalLB in cloud environments
