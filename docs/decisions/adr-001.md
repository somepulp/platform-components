# ADR-001: Use Directory-per-Environment Structure 

**Status:** Accepted

---

## Context

`platform-components` manages platform services (ArgoCD, Harbor, Vault, MetalLB, ingress-nginx, Prometheus, Grafana) across multiple environments (local, staging, prod). A strategy is needed to manage environment-specific configuration differences — for example, reduced resource limits on the Pi vs. higher limits on GCP, or different ingress hostnames per environment. Two approaches were considered: directory-per-environment and branch-per-environment.

---

## Decision

Use a directory-per-environment structure (`envs/local`, `envs/staging`, `envs/prod`) on a single main branch for environment-specific configuration overrides in `platform-components`.

---

## Reasoning

Platform services are managed exclusively by the platform team. Unlike application deployments, they change infrequently and do not require environment-specific access control gates — any platform team member can make changes across all environments. This removes the primary motivation for branch-per-environment, which is enforcing different approval requirements per environment via branch protection rules.

Directory-per-environment on a single branch offers cleaner operational visibility — all environments are visible and comparable in one place, and promoting a configuration change from local to staging is a simple file copy reviewed in a single PR rather than a cross-branch merge. It also avoids the branch drift problem where environment branches diverge over time and become difficult to reconcile.

Branch-per-environment is explicitly chosen for `platform-deploy` where developers outside the platform team interact with the repo and stronger promotion gates are needed (see `platform-deploy` ADR-001).

---

## Consequences

**Positive:**
- All environments visible and comparable in one place
- No branch drift — single source of truth
- Simpler promotion — copy files between directories, review in one PR
- Consistent with how ArgoCD Applications reference paths rather than branches for platform services

**Negative:**
- No GitHub branch protection enforcement per environment — relies on team discipline and PR review process
- A single bad merge to main could affect all environments simultaneously if config is not properly scoped to environment directories
