# ADR-002: Use Gateway API over ingress-nginx as the Ingress Solution

**Status:** Accepted

---

## Context

k3s ships with Traefik as its default ingress controller, installed automatically at cluster bootstrap. An ingress solution is required to route external HTTP/HTTPS traffic (north-south) into services running inside the cluster. The question is whether to use the bundled Traefik, replace it with ingress-nginx, or adopt the Kubernetes Gateway API.

Note: ingress controllers and Gateway API handle north-south traffic (external → cluster). East-west traffic (service → service inside the cluster) is a separate concern handled by service meshes such as Istio or Linkerd, which are out of scope for this platform at this time.

---

## Decision

Disable Traefik at k3s installation time and adopt the Kubernetes Gateway API as the cluster ingress solution, managed via `platform-components`.

---

## Reasoning

### ingress-nginx retirement

ingress-nginx was initially considered as the replacement for Traefik given its widespread production adoption. However, the Kubernetes SIG Network and Security Response Committee announced the retirement of ingress-nginx in November 2025, with best-effort maintenance continuing only until March 2026. As of the time of this decision (June 2026), ingress-nginx is past its end-of-life date — no further releases, bugfixes, or security vulnerability patches will be issued. Adopting a retired project as a foundational platform service is not appropriate even in a homelab context, as it would mean learning a tool with no future.

### Gateway API as the modern standard

The Kubernetes Gateway API is the officially recommended successor to the Ingress resource and ingress controllers. It is a CNCF standard, supported natively by GKE (the target cloud platform), and is now the direction the entire ecosystem is moving. Learning Gateway API is directly transferable to production environments at cloud-native companies.

Gateway API is more expressive than the classic Ingress spec — it separates concerns across three resource types:

- **GatewayClass** — defines the type of gateway infrastructure (e.g. which controller handles it)
- **Gateway** — defines where traffic enters the cluster (ports, protocols, TLS)
- **HTTPRoute / TCPRoute / GRPCRoute** — defines how traffic is routed to backend services

This separation makes Gateway API more composable and suitable for multi-team environments — platform teams own the Gateway, application teams own the Routes.

### Transferability

While Gateway API is the current standard, the core concepts (routing external traffic to services, TLS termination, path-based routing) are identical to those in ingress-nginx. Engineers familiar with Gateway API can function in environments still running ingress-nginx with minimal context switching, and vice versa.

---

## Alternatives Considered

| Option | Reason Not Chosen |
|---|---|
| Traefik (k3s default) | Traefik-specific CRDs reduce portability; bypasses GitOps management |
| ingress-nginx | Officially retired as of March 2026; no further security patches |
| Contour | Supports Gateway API but adds unnecessary abstraction layer |
| GKE Ingress | Cloud-native but GCP-specific; not suitable for local Pi environment |

---

## Consequences

**Positive:**
- Adopts the current and future Kubernetes networking standard
- Native GKE support — directly relevant when platform extends to GCP
- More expressive routing model than classic Ingress spec
- Explicitly managed via GitOps in `platform-components`
- Concepts transfer to ingress-nginx environments with minimal context switching

**Negative:**
- Gateway API is newer — some tutorials and community resources still reference ingress-nginx
- Requires a Gateway API-compatible controller (e.g. Envoy Gateway, Cilium, or NGINX Gateway Fabric) — adds one additional decision on controller choice
- Less homelab documentation available compared to ingress-nginx

---

## Gateway API Controller: Envoy Gateway

The Gateway API is a specification — it requires a compatible controller to implement it. The following options were evaluated:

| Controller | Reason Not Chosen |
|---|---|
| NGINX Gateway Fabric | CPU spikes when unrelated controllers are active; inefficient Kubernetes event stream filtering compared to Envoy-based implementations |
| Cilium | Requires replacing k3s's default CNI (flannel), adding significant complexity; missing TCPRoute support and forward authentication in recent real-world evaluations |
| Istio Gateway | Gateway API support trails behind Istio's native VirtualService model; overkill for a single-node homelab without service mesh requirements |

**Envoy Gateway** is chosen as the Gateway API controller for the following reasons:

- CNCF project with the most active development momentum in the Gateway API ecosystem
- Most feature-complete implementation — supports TCPRoute, BackendTLSPolicy, forward authentication, and native declarative rate limiting via CRD
- Architecturally natural fit for Gateway API — Envoy's dynamic xDS-based configuration model aligns directly with Gateway API's design philosophy
- Pi 5 has sufficient memory (8GB) to absorb Envoy Gateway's overhead
- Does not require replacing the k3s default CNI — installs alongside flannel without conflict
- Most interview-relevant — Envoy is what cloud-native teams are converging on for Gateway API implementations
