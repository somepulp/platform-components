# ADR-003: Use Harbor as the Local Container Image Registry

**Status:** Accepted

---

## Context

The platform needs a container image registry to store and serve Docker images built from application source code (e.g. `ml-classifier`). In a cloud environment this would typically be a managed registry service (GCP Artifact Registry, AWS ECR, Docker Hub). For local development on the Raspberry Pi, a self-hosted registry is preferred to avoid cloud costs, external dependencies, and to keep the full ML platform lifecycle local and self-contained.

---

## Decision

Deploy Harbor as the self-hosted container image registry, managed via `platform-components`.

---

## Reasoning

Harbor is a CNCF graduated project and the most widely adopted open source container registry in production environments. It goes significantly beyond a basic registry — it provides image vulnerability scanning, role-based access control (RBAC), image replication, and a web UI for image management.

Running Harbor locally means the full image lifecycle (build → push → pull → deploy) stays within the homelab without any external dependencies or costs. For the `ml-classifier` service specifically, Harbor allows the CI pipeline (GitHub Actions) to push built images to a local registry, which ArgoCD and Kubernetes then pull from during deployment. This completes the GitOps loop entirely on local infrastructure.

When the platform extends to GCP, Harbor can either be retained as a local registry with replication to GCP Artifact Registry, or replaced entirely by a cloud-native registry. That decision is deferred until cloud environments are provisioned.

---

## Alternatives Considered

| Registry | Reason Not Chosen |
|---|---|
| Docker Hub | External dependency, rate limits on pulls, requires internet access |
| GCP Artifact Registry | Cloud cost, external dependency, not available for local-only setup |
| AWS ECR | Cloud cost, external dependency, not aligned with GCP-first cloud strategy |
| Docker Registry (basic) | No UI, no RBAC, no vulnerability scanning — too bare for a production-credible platform |
| Gitea with built-in registry | Adds unnecessary complexity; primary need is a registry, not a Git server |

---

## Consequences

**Positive:**
- CNCF graduated project — production-credible and widely used in enterprise environments
- Full image lifecycle stays local — no cloud costs or external dependencies
- Built-in vulnerability scanning, RBAC, and web UI
- Replication support enables future migration path to cloud registries
- Directly relevant to the skills being built for ML Platform Engineer roles

**Negative:**
- Heavier than a basic registry — requires more Pi resources (memory, storage)
- Adds operational complexity — Harbor itself needs to be maintained and monitored
- Requires TLS configuration for secure image pulls from Kubernetes
