# ADR-003: Use Zot as the Local Container Image Registry

**Status:** Accepted (supersedes original Harbor decision)

---

## Context

The platform needs a container image registry to store and serve Docker images built from application source code (e.g. `ml-classifier`). The original decision was to use Harbor as the self-hosted registry. However, Harbor was found to be incompatible with the Raspberry Pi 5 (ARM64/aarch64) architecture during implementation.

---

## Decision

Deploy Zot as the self-hosted container image registry for the local Pi environment, managed via `platform-components`. Use GCP Artifact Registry for staging and production environments on GCP.

---

## Reasoning

### Why a self-hosted registry

Running a self-hosted registry locally means the full image lifecycle (build → push → pull → deploy) stays within the homelab without any external dependencies or costs. For the `ml-classifier` service specifically, the CI pipeline (GitHub Actions) pushes built images to the local registry, which ArgoCD and Kubernetes then pull from during deployment. This completes the GitOps loop entirely on local infrastructure.

### Why not Docker Hub or other external registries

- Docker Hub has rate limits on pulls (100 pulls/6 hours for free tier)
- Requires internet access for every image pull — creates an external dependency for every pod start
- Images stored externally — breaks the self-contained local platform story
- Free tier only allows 1 private repository

### Why Harbor was rejected

Harbor is a CNCF graduated project and the most widely adopted open source container registry in production environments. It was the original registry choice for this platform. However, the official Harbor images published at `docker.io/goharbor/` are built for `amd64` only. Attempting to deploy Harbor on the Raspberry Pi 5 (aarch64) results in `exec format error` on init containers — the images cannot run on ARM64.

An ARM64 support PR exists ([goharbor/harbor#21825](https://github.com/goharbor/harbor/pull/21825)) but is stalled — Harbor maintainers requested a formal proposal and CI/release process involvement before merging, with no timeline committed. ARM64 support is not available in official Harbor releases.

### Why Zot

Zot is a CNCF sandbox project and OCI-native container registry designed for simplicity and resource-constrained environments. Key reasons for choosing Zot for the local Pi environment:

- **ARM64 native** — Zot publishes official multi-arch images including `linux/arm64`. All images support multiple architectures (linux/amd64, linux/arm64, freebsd/amd64, freebsd/arm64)
- **Lightweight** — minimal resource footprint, well-suited for a single-node Pi cluster alongside other platform services
- **OCI-compliant** — fully implements the OCI Distribution Specification, compatible with Docker, Helm, and any OCI client
- **No external dependencies** — no database required, stores data on local filesystem via `local-path-provisioner`
- **Web UI included** — the `ui` extension is built into the binary and enabled by default. Provides image browsing, search, vulnerability scan results, and pull commands
- **CNCF sandbox** — actively maintained, production-credible

### Why GCP Artifact Registry for staging/prod

When the platform extends to GCP, GCP Artifact Registry is the natural registry choice — it is the native managed registry for GKE, integrates directly with GCP IAM and Cloud Build, requires no operational overhead, and eliminates the need to self-host a registry in cloud environments. There is no reason to run a self-hosted registry on GCP when a managed alternative exists.

---

## Alternatives Considered

| Registry | Reason Not Chosen |
|---|---|
| Harbor | Official images are amd64-only; ARM64 support PR stalled with no timeline |
| Docker Hub | External dependency, rate limits, breaks self-contained local platform story |
| JFrog Artifactory OSS | Minimum 4 CPU cores and 6GB RAM — too resource intensive for Pi alongside other platform services |
| GCP Artifact Registry (local) | Cloud dependency, not available for offline local Pi cluster |

---

## Consequences

**Positive:**
- ARM64 compatible — runs natively on Raspberry Pi 5
- Lightweight — low resource footprint suitable for Pi alongside ArgoCD, Vault, Envoy Gateway, cert-manager
- OCI-compliant — skills and tooling transfer directly to Harbor or GCP Artifact Registry environments
- Self-contained — no external dependencies or cloud costs for local development
- Full image lifecycle stays local — completes the GitOps loop entirely on local infrastructure
- Web UI included by default

**Negative:**
- Less feature-rich than Harbor — no built-in vulnerability scanning (Trivy integration is available but requires separate configuration), no image replication built-in
- Web UI (`zui`) is available but less mature than Harbor's portal
- Zot is a CNCF sandbox project (not graduated) — less mature than Harbor
- Different registry per environment (Zot local, GCP Artifact Registry cloud) — adds environment-specific configuration complexity
