# ADR-006: ArgoCD Repository Authentication and Credential Bootstrapping

**Status:** Accepted

---

## Context

ArgoCD requires SSH credentials to clone and watch private GitHub repositories under the `somepulp` organization. A decision is needed on:

1. How to scope SSH credentials — a single account-level key vs. per-repo deploy keys
2. Where private keys are stored at rest
3. How to handle the bootstrapping chicken-and-egg problem: keys should live in Vault, but Vault is installed by ArgoCD, which needs keys to access repos before Vault exists

---

## Decision

Use per-repo GitHub deploy keys scoped to individual repositories. Private keys are stored in HashiCorp Vault as the steady-state. During initial bootstrap, a single temporary deploy key for `platform-components` is stored as a Kubernetes Secret and migrated to Vault once Vault is operational.

---

## Reasoning

### Per-repo deploy keys over a single account-level SSH key

A single account-level SSH key grants ArgoCD access to all repositories the GitHub account can access. If the key is compromised, the blast radius covers every repo. Per-repo deploy keys follow the principle of least privilege — each key grants access to exactly one repository, so a compromised key exposes only that repo.

GitHub deploy keys are read-only by default, which is appropriate for ArgoCD — it only needs to read from repos, never write to them.

The operational overhead of managing 5 key pairs is justified by the security boundary each key creates. In production environments, this overhead is further reduced by automating key rotation via Vault and the GitHub API.

### Vault as the steady-state credential store

Private keys stored on a developer's laptop are a security liability — laptops can be lost, stolen, or compromised. In production, private keys should be generated, immediately stored in Vault, and never persist on a local machine long-term.

Vault provides:
- Encrypted storage for all private keys
- Audit trail of every secret access
- TTL-based expiry and rotation policies
- ArgoCD retrieves credentials from Vault at runtime via the Vault Secrets Operator

### Bootstrapping sequencing

The chicken-and-egg problem: ArgoCD needs credentials to access `platform-components` to install Vault, but Vault doesn't exist yet to store those credentials.

The resolution is a phased credential migration:

**Phase 1 — Bootstrap (temporary):**
- Generate one deploy key for `platform-components` only
- Store the private key as a Kubernetes Secret in the `argocd` namespace
- Add the public key to `platform-components` as a GitHub deploy key
- ArgoCD uses this key to sync `platform-components` and install platform services including Vault

**Phase 2 — Migrate to Vault (steady-state):**
- Generate the remaining 4 deploy keys (`platform-infra`, `platform-charts`, `platform-deploy`, `ml-classifier`)
- Store all 5 private keys in Vault
- Migrate the `platform-components` key from the Kubernetes Secret to Vault
- Register all repos with ArgoCD using Vault-sourced credentials
- Delete the temporary Kubernetes Secret

This phased approach is a well-established pattern for GitOps bootstrapping — one imperative step enables the declarative system, which then manages everything going forward including its own credentials.

## Consequences

**Positive:**
- Principle of least privilege — each key scoped to one repo, blast radius minimized per key
- Vault as steady-state means no credentials persist on developer machines long-term
- Phased migration provides a pragmatic path through the bootstrapping chicken-and-egg problem
- Deploy keys are read-only by default — ArgoCD cannot accidentally push to repos

**Negative:**
- 5 key pairs to manage, generate, and rotate rather than one
- Bootstrap phase requires a temporary Kubernetes Secret — must be deleted after migration to Vault
- Key rotation requires coordinated updates in both Vault and GitHub
